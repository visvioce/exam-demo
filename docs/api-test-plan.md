# 考试系统 API 角色化测试方案

> 基于 `docs/hoppscotch-collection.json` 中定义的接口，按角色组织完整测试用例。
> 本文档中所有请求体均为自行构造，不使用 collection 中的原始数据。

---

## 一、测试环境

| 配置项 | 值 |
|--------|-----|
| Base URL | `http://localhost:8080` |
| Content-Type | `application/json` |
| 鉴权方式 | Bearer Token（Header: `Authorization: Bearer <token>`） |

---

## 二、角色定义与 Token 变量

| 角色 | 角色标识 | Token 变量 | 说明 |
|------|---------|-----------|------|
| 管理员 | `ADMIN` | `<<adminToken>>` | 系统最高权限，可管理用户、轮播图等 |
| 教师 | `TEACHER` | `<<teacherToken>>` | 可管理题库、试卷、考试、课程、公告、AI 配置 |
| 学生 | `STUDENT` | `<<studentToken>>` | 可参加考试、加入课程、查看已发布考试 |
| 匿名用户 | 无 | 无 Token | 仅可访问登录/注册等公开接口 |

### Token 获取方式

测试前需先通过以下流程获取各角色 Token：

1. **管理员 Token**：`POST /api/auth/login` → `{"username":"admin_user","password":"Admin@123456"}` → 提取响应中的 `token`
2. **教师 Token**：`POST /api/auth/login` → `{"username":"teacher_user","password":"Teacher@123456"}` → 提取响应中的 `token`
3. **学生 Token**：`POST /api/auth/login` → `{"username":"student_user","password":"Student@123456"}` → 提取响应中的 `token`

> 若上述账号不存在，需先通过注册接口创建或由管理员通过 `POST /api/users` 创建。

## 三、通用约定

### 3.1 "删除前先创建"流程

所有 DELETE 类接口测试严格遵循以下流程：

```
步骤 1：以有权限角色调用 POST 接口创建测试数据 → 记录返回的 ID
步骤 2：以有权限角色调用 DELETE 接口删除该 ID
步骤 3：以有权限角色调用 GET 接口验证数据已不存在（期望 404 或列表中不含该数据）
```

### 3.2 权限校验矩阵

| 角色 | 用户管理增删改 | 题库/试卷增删改 | 考试增删改 | 课程增删改 | 公告增删改 | 轮播图增删改 | AI 配置管理 |
|------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| 管理员 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 教师 | ❌ 403 | ✅ | ✅ | ✅ | ✅ | ❌ 403 | ✅ |
| 学生 | ❌ 403 | ❌ 403 | ❌ 403 | ❌ 403 | ❌ 403 | ❌ 403 | ❌ 403 |
| 匿名 | ❌ 401 | ❌ 401 | ❌ 401 | ❌ 401 | ❌ 401 | ❌ 401 | ❌ 401 |

### 3.3 预期的 HTTP 状态码

| 场景 | 状态码 |
|------|:-----:|
| 查询成功 | 200 |
| 创建成功 | 201 |
| 更新成功 | 200 |
| 删除成功 | 200 或 204 |
| 未认证（无 Token） | 401 |
| 无权限（越权访问） | 403 |
| 资源不存在 | 404 |
| 参数校验失败 | 400 |
| 业务逻辑冲突 | 409 |

### 3.4 请求体说明

- 所有测试用例中的请求体均为自行构造
- 未直接使用 hoppscotch-collection.json 中的 body 内容
- 测试数据使用 `test_` 前缀或明确的测试标识便于识别与清理

> ```bash

>   -H "Authorization: Bearer <<adminToken>>" \
>   -H "Content-Type: application/json" \

>   -d '{"subject":"数学","type":"SINGLE_CHOICE","difficulty":"MEDIUM","count":3,"requirements":"考察函数知识点"}'
> ```
> 预期：终端持续输出 `data:` 开头的 SSE 事件，直至流结束。

## 五、测试执行建议

### 5.1 执行顺序

1. **认证管理**：先执行登录/注册获取各角色 Token
2. **课程管理**：创建测试课程（被考试、试卷依赖）
3. **题库管理**：创建测试题目（被试卷依赖）
4. **试卷管理**：创建测试试卷（被考试依赖）
5. **考试管理**：执行完整生命周期测试
6. **考试记录管理**：依赖考试数据
7. **用户管理**：独立模块，随时可测
8. **公告管理**：独立模块
9. **轮播图管理**：独立模块
10. **AI 配置管理**：独立模块
11. **AI 出题**：依赖 AI 配置

### 5.2 测试数据清理

所有 DELETE 测试均已在流程中验证删除，无需额外清理。但以下情况需注意：

- 若创建测试数据后测试中断，需手动清理残留数据
- 批量删除测试中的用户需确保所有前置步骤都完成
- AI 配置中的 API Key 建议使用测试用无效 Key，避免误用
