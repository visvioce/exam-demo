## 一、架构相关术语

### 1. B/S 架构

- **论文位置**：1.1 研究背景与意义、2.1.1 技术可行性分析
- **论文描述**：基于B/S架构的在线考试系统具有维护成本低、客户端无需安装等优势。
- **项目应用**：
  - 系统采用浏览器-服务器模式。前端Vue3应用运行在浏览器端，后端Spring Boot服务运行在服务器端，用户无需安装任何客户端软件，通过浏览器即可访问所有功能。前端通过HTTP请求与后端RESTful API交互。
- **答辩追问**：为什么不用C/S架构？
  - **回答要点**：C/S需要安装客户端，维护成本高（更新要重新分发安装包）；B/S零安装、跨平台，适合学校机房、学生个人电脑等多种环境。考试系统面向大量学生，B/S是更合理的选择。

### 2. 前后端分离架构

- **论文位置**：1.3 开发技术、1.5 研究方法、2.5.4 可维护性需求
- **论文描述**：系统划分为表现层、业务逻辑层、数据访问层三层，前后端通过JSON格式交互，降低耦合度，便于独立开发和部署。
- **项目应用**：
  - 前端项目：`/frontend` 目录，基于 Vite + Vue3 + TypeScript + Element Plus
  - 后端项目：`/backend` 目录，基于 Spring Boot + MyBatis-Plus
  - 前后端完全独立开发和部署，通过HTTP + JSON进行RESTful API数据交互
  - CORS跨域配置在[SecurityConfig.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/config/SecurityConfig.java#L53-L66)
- **答辩追问**：前后端分离有什么好处？有什么缺点？
  - **回答要点**：
    - 好处：①前后端可并行开发，提高团队效率；②前端可以独立部署和优化；③后端API可被多端复用（如后续开发移动端）。
    - 缺点：①首次加载需要下载前端资源，首屏稍慢（可用懒加载优化）；②SEO不友好（考试系统后台管理不需要SEO，不影响）；③需要处理跨域问题（本项目通过CORS配置解决）。

### 3. 三层架构

- **论文位置**：1.3 开发技术
- **论文描述**：系统分为表现层、业务逻辑层、数据访问层三层
- **项目应用**：
  - 表现层（Controller）：`controller/` 包，如`ExamController.java`、`UserController.java`，负责接收HTTP请求、参数校验、返回JSON响应
  - 业务逻辑层（Service）：`service/` 包，如`ExamService.java`、`UserService.java`，负责业务规则处理、事务管理
  - 数据访问层（Mapper）：`mapper/` 包，基于MyBatis-Plus的BaseMapper，负责数据库CRUD操作
- **答辩追问**：三层架构和MVC有什么区别？
  - **回答要点**：三层架构是纵向分层（表现层→业务层→数据层），MVC是表现层内部的横向划分（Model-View-Controller）。本项目Controller对应MVC的C，Service+Entity对应M，Vue前端对应V。三层架构更关注整个应用的分层，MVC更关注表现层内部的职责划分。

## 二、后端技术

### 4. Spring Boot

- **论文位置**：1.3 开发技术
- **论文描述**：通过自动配置、起步依赖等特性简化Spring应用开发，提供IoC容器、声明式事务管理、Web框架集成和数据持久化接口。
- **项目应用**：
  - 版本：Spring Boot 3.3.5，见[pom.xml](file:///home/xigua/work/exam-demo/backend/pom.xml#L11)
  - 启动类：[ExamApplication.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/ExamApplication.java#L7)
  - 配置文件：[application.yml](file:///home/xigua/work/exam-demo/backend/src/main/resources/application.yml)
- **答辩追问**：Spring Boot的自动配置原理是什么？
  - **回答要点**：Spring Boot通过`@EnableAutoConfiguration`注解，根据classpath中的jar包依赖和配置文件，自动配置Spring应用。例如引入`spring-boot-starter-web`就自动配置Tomcat和Spring MVC；引入`spring-boot-starter-data-jpa`就自动配置数据源和JPA。开发者只需在`application.yml`中覆盖默认配置即可。

### 5. IoC 容器（控制反转）

- **论文位置**：1.3 开发技术
- **论文描述**：SpringBoot提供IoC容器
- **项目应用**：
  - Spring IoC容器管理所有Bean的生命周期和依赖注入
  - Service类通过`@Service`注解声明为Bean
  - 依赖注入通过构造器注入实现（`@RequiredArgsConstructor`），如[ExamService.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/service/ExamService.java#L48-L49)：
    ```java
    @RequiredArgsConstructor
    public class ExamService {
        private final PaperService paperService;
        private final QuestionService questionService;
        // ...
    }
    ```
- **答辩追问**：为什么用构造器注入而不是字段注入？
  - **回答要点**：①构造器注入的依赖不可变（final），保证对象创建时所有依赖已就绪；②便于单元测试，可以直接传入Mock对象；③避免循环依赖问题；④Lombok的`@RequiredArgsConstructor`可以自动生成构造器，代码更简洁。

### 6. 声明式事务管理

- **论文位置**：1.3 开发技术
- **论文描述**：Spring提供的声明式事务管理
- **项目应用**：
  - 通过`@Transactional`注解声明事务，如[ExamService.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/service/ExamService.java#L135-L136)：
    ```java
    @Transactional
    public boolean createExam(ExamCreateRequest examRequest, Long teacherId) {
    ```
  - 创建考试、提交考试、评分等涉及多表操作的业务均使用事务管理，确保ACID特性
  - 定时任务使用`@Transactional(propagation = Propagation.REQUIRES_NEW)`开启新事务
- **答辩追问**：`@Transactional`的原理是什么？什么情况下会失效？
  - **回答要点**：原理是通过AOP代理，在方法执行前后自动开启/提交/回滚事务。失效场景：①同类方法内部调用（this调用不走代理）；②方法不是public；③异常被catch住没有抛出；④数据库引擎不支持事务（如MyISAM）。

### 10. 响应式数据绑定

- **论文位置**：2.1.1 技术可行性分析
- **论文描述**：实现响应式数据绑定
- **项目应用**：
  - Vue 3的响应式系统：`ref()`用于基本类型，`reactive()`用于对象
  - `computed()`计算属性和`watch()`侦听器实现数据联动
  - Pinia状态管理库提供全局响应式store
- **答辩追问**：Vue3的响应式原理是什么？
  - **回答要点**：Vue3使用ES6的Proxy对象拦截数据的get/set操作。当读取响应式数据时，收集依赖（哪些组件/计算属性使用了这个值）；当修改数据时，触发所有依赖的更新。相比Vue2的Object.defineProperty，Proxy可以监听新增属性、数组索引变化等。

## 四、数据库相关

### 13. MySQL 8.0（关系型数据库）

- **论文位置**：1.3 开发技术、3.2 数据库物理结构
- **论文描述**：开源、跨平台、高性能，支持主键、外键等完整性约束。
- **项目应用**：
  - [application.yml](file:///home/xigua/work/exam-demo/backend/src/main/resources/application.yml#L19)中配置MySQL数据源
  - [schema.sql](file:///home/xigua/work/exam-demo/database/schema.sql)中定义了10张核心数据表
  - 所有表使用InnoDB引擎，支持事务
- **答辩追问**：为什么选MySQL而不是PostgreSQL或MongoDB？
  - **回答要点**：①MySQL在国内使用最广泛，文档和社区资源丰富；②学校教学也以MySQL为主，团队熟悉；③考试系统的数据关系明确（学生-考试-成绩），关系型数据库更合适；④MongoDB适合非结构化数据，本系统的数据结构化程度高。

### 14. ACID 事务

- **论文位置**：3.2 数据库物理结构
- **论文描述**：MySQL具备成熟的ACID事务机制
- **项目应用**：
  - 通过Spring的`@Transactional`注解自动管理数据库事务
  - 考试创建、提交、评分等操作均保证事务原子性
  - 数据库引擎使用InnoDB（`ENGINE=InnoDB`），支持行级锁和事务回滚
- **答辩追问**：ACID分别代表什么？举一个本系统的例子。
  - **回答要点**：A原子性（Atomicity）：事务要么全成功要么全回滚；C一致性（Consistency）：事务前后数据完整性约束不被破坏；I隔离性（Isolation）：并发事务互不干扰；D持久性（Durability）：事务提交后数据永久保存。例子：创建考试时，如果试卷题目复制成功但考试基本信息保存失败，整个事务回滚，不会出现"有题目但没有考试"的不一致状态。

### 15. MyBatis-Plus

- **论文位置**：1.3 开发技术、5.4.4 安全性测试
- **论文描述**：MyBatis的增强工具，提供通用CRUD操作、条件构造器、分页插件等特性。
- **项目应用**：
  - 版本3.5.7，见[pom.xml](file:///home/xigua/work/exam-demo/backend/pom.xml#L23)
  - 配置在[MyBatisPlusConfig.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/config/MyBatisPlusConfig.java)
  - Service层继承`ServiceImpl<Mapper, Entity>`，获得CRUD能力
  - 使用Lambda查询条件构造器实现类型安全的查询，如
    ```java
    lambdaQuery().eq(Exam::getStatus, status).list()
    ```
- **答辩追问**：MyBatis-Plus和JPA/Hibernate相比有什么优缺点？
  - **回答要点**：MyBatis-Plus优点：①SQL可控，复杂查询可以手写SQL；②学习曲线低，会MyBatis就能用；③性能更好，没有Hibernate的N+1问题。缺点：①需要写Mapper接口（虽然BaseMapper提供了很多）；②没有Hibernate的自动建表功能。本系统选择MyBatis-Plus是因为需要灵活控制SQL，且团队更熟悉MyBatis。

### 16. Lambda 语法（Lambda查询条件）

- **论文位置**：1.3 开发技术
- **论文描述**：通过Base Mapper接口提供基础增删改查，支持Lambda语法构建类型安全的查询条件。
- **项目应用**：
  - 到处使用Lambda表达式构建查询条件，如[ExamService.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/service/ExamService.java#L67)：
    ```java
    lambdaQuery().eq(Exam::getCourseId, courseId).list()
    ```
  - 避免字符串硬编码列名，编译器检查，减少字段名写错导致的运行时错误
- **答辩追问**：Lambda查询和普通Wrapper查询有什么区别？
  - **回答要点**：Lambda查询使用`Exam::getCourseId`这样的方法引用，编译时会检查字段是否存在，如果字段改名IDE会报错。普通Wrapper使用字符串`"course_id"`，字段改名后不会报错，运行时才出错。Lambda查询类型安全，是推荐做法。

### 17. 条件构造器

- **论文位置**：1.3 开发技术
- **论文描述**：支持Lambda语法构建类型安全的查询条件。
- **项目应用**：
  - `LambdaQueryWrapper`构建复杂查询条件，如[ExamService.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/service/ExamService.java#L292-L298)：
    ```java
    LambdaQueryWrapper<Exam> wrapper = new LambdaQueryWrapper<>();
    wrapper.in(Exam::getCourseId, courseIds)
            .notIn(Exam::getStatus, "DRAFT", "ENDED")
            .ge(Exam::getEndedAt, LocalDateTime.now());
    ```
- **答辩追问**：如果查询条件特别复杂，MyBatis-Plus搞不定怎么办？
  - **回答要点**：MyBatis-Plus支持在Mapper中写原生SQL（`@Select`、`@Insert`等注解），复杂查询可以手写SQL。本系统的统计查询（如成绩分布）就使用了手写SQL，见[ExamMapper.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/mapper/ExamMapper.java)。

### 18. 逻辑删除

- **论文位置**：2.3 数据模型（数据表设计）、3.2 数据库物理结构
- **论文描述**：使用`deleted`字段标识，0=正常，1=已删除
- **项目应用**：
  - 大多数核心表（users、courses、exams、questions、exam_sessions、announcements、carousels）都有`deleted`字段
  - 例外：`papers`（试卷表）和`course_members`（选课表）不使用逻辑删除，执行物理删除
  - MyBatis-Plus配置在[application.yml](file:///home/xigua/work/exam-demo/backend/src/main/resources/application.yml#L40-L43)：
    ```yaml
    logic-delete-field: deleted
    logic-delete-value: 1
    logic-not-delete-value: 0
    ```
  - 实体类中使用`@TableLogic`注解，如[ExamSession.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/entity/ExamSession.java#L75-L76)
  - 执行delete操作时自动转换为update set deleted=1，查询时自动过滤deleted=0的记录
- **答辩追问**：为什么考试记录要用逻辑删除？
  - **回答要点**：考试记录涉及学生成绩，是重要数据，不能物理删除。如果误删无法恢复。逻辑删除保留了历史数据，便于成绩追溯和审计。同时MyBatis-Plus自动处理逻辑删除，对业务代码透明，不需要手动加`WHERE deleted=0`条件。

### 19. 数据库索引

- **论文位置**：1.3 开发技术
- **论文描述**：规范的表结构设计和索引优化能充分发挥查询效率
- **项目应用**：
  - 在[schema.sql](file:///home/xigua/work/exam-demo/database/schema.sql#L171-L199)中创建了19个索引
  - 覆盖高频查询字段：`exam_sessions(exam_id, student_id, status)`
  - 复合索引：`idx_exam_sessions_exam_status (exam_id, status)`
  - 时间范围索引：`idx_exams_time_range (started_at, ended_at)`
- **答辩追问**：索引是不是越多越好？
  - **回答要点**：不是。索引会占用额外存储空间，且每次插入/更新/删除时需要维护索引，降低写性能。应该只为高频查询字段创建索引。本系统19个索引覆盖了最常用的查询场景，如按考试ID查学生记录、按状态查考试等。

### 20. 外键约束

- **论文位置**：1.3 开发技术
- **论文描述**：支持主键、外键等完整性约束
- **项目应用**：
  - [schema.sql](file:///home/xigua/work/exam-demo/database/schema.sql)中定义了多对外键约束
  - 如：`exam_sessions.exam_id → exams.id`、`exam_sessions.student_id → users.id`
  - 保证引用完整性，防止孤儿数据
- **答辩追问**：外键有什么优缺点？
  - **回答要点**：优点：保证数据一致性，防止删除被引用的记录。缺点：影响性能（每次操作都要检查外键），高并发场景可能成为瓶颈。本系统使用外键是因为数据一致性要求高（如不能删除已有关联考试的试卷），且并发量不大。

### 21. 唯一索引 / 复合唯一索引

- **论文位置**：3.2 数据库物理结构（course_members表说明）
- **论文描述**：通过复合唯一索引确保学生选课的唯一性
- **项目应用**：
  - 选课表：[schema.sql](file:///home/xigua/work/exam-demo/database/schema.sql#L57)：`UNIQUE KEY uk_course_student (course_id, student_id)`
  - 考试记录表：[schema.sql](file:///home/xigua/work/exam-demo/database/schema.sql#L123)：`UNIQUE KEY uk_exam_student (exam_id, student_id)`
  - 同一学生不能重复选课、不能重复参加同一考试
- **答辩追问**：唯一索引和主键有什么区别？
  - **回答要点**：主键是特殊的唯一索引，一个表只能有一个主键，主键列不能为NULL。唯一索引可以有多个，允许NULL值（但只能有一个NULL）。本系统中`exam_sessions`表的主键是自增ID，`(exam_id, student_id)`是唯一索引，保证业务层面的唯一性。

## 五、安全相关术语

### 22. JWT 认证

- **论文位置**：摘要、1.5 研究方法、2.2.2 需求描述、5.4.4 安全性测试
- **论文描述**：JWT身份验证，作为身份凭证进行用户身份校验
- **项目应用**：
  - JWT配置：[application.yml](file:///home/xigua/work/exam-demo/backend/src/main/resources/application.yml#L47-L49)中配置secret和过期时间（7天）
  - JWT拦截器：[JwtAuthenticationFilter.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/config/JwtAuthenticationFilter.java) 拦截请求验证Token
  - Spring Security配置：[SecurityConfig.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/config/SecurityConfig.java#L38) 设置为无状态（`STATELESS`），每请求验证Token
  - 登录接口返回JWT Token，前端存储在localStorage，每次请求携带在Authorization header
- **答辩追问**：JWT和传统Session认证有什么区别？
  - **回答要点**：
    - Session：服务端存储会话状态，客户端存Session ID，有状态，不适合分布式。
    - JWT：服务端不存储状态，Token自包含用户信息，无状态，适合分布式和前后端分离。
    - 本项目选择JWT是因为前后端分离架构，且需要支持Token刷新机制。
- **答辩追问**：JWT有什么安全问题？怎么解决？
  - **回答要点**：①Token被盗用：设置合理过期时间（7天），支持Token刷新；②Token无法主动失效：结合黑名单或缩短过期时间；③密钥泄露：密钥通过环境变量注入，不写在代码中。本系统实现了Token刷新机制，过期前可续期。

### 24. BCrypt 加密

- **论文位置**：2.5.3 安全需求、5.4.4 安全性测试
- **论文描述**：用户密码采用BCrypt加密存储
- **项目应用**：
  - [PasswordUtil.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/utils/PasswordUtil.java)：
    ```java
    private static final BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();
    ```
  - 密码存储为BCrypt哈希值，不可逆，即使数据库泄露也无法推算明文密码
  - 数据库种子数据中的密码`123456`存储为BCrypt加密后的值
- **答辩追问**：为什么用BCrypt而不是MD5？
  - **回答要点**：MD5是哈希算法，速度快，容易被暴力破解（彩虹表攻击）。BCrypt是自适应哈希算法，内置salt（随机盐值），且可以通过调整work factor增加计算时间，抵抗暴力破解。Spring Security官方推荐BCrypt作为密码加密方案。

### 26. Spring Security

- **论文位置**：2.5.3 安全需求、5.4.4 安全性测试

### 27. SQL注入防护

- **论文位置**：5.4.4 安全性测试
- **论文描述**：后端基于MyBatis-Plus进行参数化数据访问，降低了SQL注入风险
- **项目应用**：
  - MyBatis-Plus内置参数化查询（预编译语句PreparedStatement），所有查询条件通过占位符`?`传递
  - 使用Lambda查询构造器，条件值自动参数化，不直接拼接SQL字符串
  - 未使用任何拼接SQL或字符串模板的方式构建查询
- **答辩追问**：参数化查询为什么能防止SQL注入？
  - **回答要点**：参数化查询将SQL语句和参数分开传递，数据库先将SQL语句编译成执行计划，然后再传入参数。参数被视为纯数据，不会被解析为SQL语法。即使参数中包含`' OR 1=1 --`这样的恶意代码，也只会被当作普通字符串处理。

### 28. XSS攻击防护

- **论文位置**：5.4.4 安全性测试
- **论文描述**：前端对部分展示内容进行基础净化与转义处理
- **项目应用**：
  - Vue3默认对`{{ }}`插值表达式中的内容进行HTML转义
  - Element Plus组件对输入内容进行基础处理
- **答辩追问**：XSS攻击的原理是什么？
  - **回答要点**：XSS（跨站脚本攻击）是攻击者在输入中注入恶意JavaScript代码，当其他用户浏览页面时，浏览器执行了这段代码。例如用户在评论中输入`<script>alert(document.cookie)</script>`，如果没有转义，这段代码会被执行。Vue3默认对插值内容进行HTML实体编码，`<`变成`&lt;`，浏览器会显示为文本而不是执行。

## 六、并发控制

### 29. 乐观锁

- **论文位置**：摘要、2.2.2 需求描述、2.5.2 可靠性需求、3.2 数据库物理结构（exam_sessions表）
- **论文描述**：在考试提交与成绩更新等关键环节引入乐观锁机制，降低并发提交带来的数据冲突风险
- **项目应用**：
  - MyBatis-Plus配置：[MyBatisPlusConfig.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/config/MyBatisPlusConfig.java#L18-L19)注册乐观锁拦截器
  - 实体类：[ExamSession.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/entity/ExamSession.java#L37-L38)：
    ```java
    @Version
    private Integer version;
    ```
  - 数据库：[schema.sql](file:///home/xigua/work/exam-demo/database/schema.sql#L119)：`version INT NOT NULL DEFAULT 0`
  - 异常处理：[GlobalExceptionHandler.java](file:///home/xigua/work/exam-demo/backend/src/main/java/com/southcollege/exam/exception/GlobalExceptionHandler.java#L164-L172)统一捕获`OptimisticLockingFailureException`
  - 应用场景：自动保存、提交考试、教师评分、批量阅卷四个关键操作都捕获了乐观锁异常
- **答辩追问**：乐观锁和悲观锁有什么区别？
  - **回答要点**：
    - 乐观锁：假设冲突少，更新时检查版本号，冲突时重试或报错。适合读多写少场景。
    - 悲观锁：假设冲突多，读取时就加锁，其他人等待。适合写多读少场景。
    - 本系统选择乐观锁是因为考试提交是低频操作（一个学生只提交一次），冲突概率低，乐观锁性能更好。
- **答辩追问**：乐观锁在本系统中具体防止什么问题？
  - **回答要点**：主要防止重复提交。例如学生快速双击提交按钮，两个请求同时到达，都发现考试未提交（第626行检查通过），然后都执行update。乐观锁保证只有一个请求能成功更新version，另一个抛出异常，返回"已提交，请勿重复提交"。

- **论文位置**：摘要、1.3 开发技术、3.2 数据库物理结构
